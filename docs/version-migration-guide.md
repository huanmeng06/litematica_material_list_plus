# LMLP 新版本迁移指南

最后更新：2026-07-15

本文集中记录 1.20.1～26.1.2 适配时实际发生过的问题。迁移新版本前，先更新
`versions.toml`，再按本文检查，不要直接复制相邻版本后只处理编译错误。

## 固定流程

1. 登记 Minecraft、Java、Fabric API、Litematica、MaLiLib、JEI 和 Mod Menu 版本。
2. 创建独立 `mc...` 分支和 worktree。
3. 先让 Gradle 构建和依赖下载可复现，再迁移源码。
4. 按基础类型、GUI、JEI、Mixin、世界渲染的顺序处理 API 差异。
5. 执行 `tools/lmlp build`、`verify` 和 `regression-check`。
6. 部署后确认实例中只有一个有效 LMLP JAR，并核对 SHA-256。
7. 游戏内验证通过后再提交、推送和准备新 Release。

## 构建与命名空间

- 1.20.1/1.20.4 成品为 Java 17（class major 61），但当前 Loom 构建进程需要 Java 21。
- 1.20.6～1.21.11 为 Java 21（class major 65）。
- 26.1.2 成品目标为 Java 25（class major 69），当前使用 Java 26 构建。
- 26.1+ 使用 Mojang 官方名称，不再使用 `class_`、`method_`、`field_` intermediary 名称。
- Mixin 字符串目标不会被 Java 编译器完整验证，必须使用目标 JAR 和 `javap` 核对描述符。
- 26.1.2 的 Java TLS 与 Modrinth CDN 曾发生握手失败；依赖由管理工具先下载到
  `.lmlp/dependencies`，Gradle 再使用本地缓存。

## 热键重置

- 历史表现：绑定后重置按钮不立即亮起，点击搜索框后才刷新。
- `initGui()` 必须先清理旧监听器，再由基类创建新列表和监听器；顺序反过来会清掉新监听器。
- 绑定完成后只刷新 LMLP 自己的热键按钮，不向 MaLiLib 公共配置页面注入全局逻辑。
- 保留 MaLiLib 原生 `IKeybind.isModified()` 判断。
- 不要为了刷新一个按钮重建或清空整个配置列表；曾导致页面只剩搜索图标。

## 原点光束和渲染层级

- 使用对应版本原版 `beacon_beam` 纹理、几何和 UV，不恢复旧自绘方柱。
- 光束为红色 OMMC 样式，关闭深度测试和深度写入，必须穿实体方块。
- 光束应在蓝图、云层、天气和粒子之上；靶心与标签最后绘制。
- 1.21.1 使用最终世界渲染事件和明确的 no-depth layer；普通 WorldRenderer TAIL 曾导致光束消失。
- 1.21.10/1.21.11 使用 FrameGraph，必须在完整 FrameGraph 执行后绘制。
- 26.1.2 的 RenderPipeline、DepthStencilState 和渲染回调描述符均已变化，不能照搬类名。
- 标签保持红框黑底，且仅在注视目标时出现；靶心和原点文字不能丢失。

## GUI、HUD 和列表

- 26.1.2 使用 GUI 提取阶段；`render`、`GuiGraphics` 等旧入口不能只靠改名猜测。
- 列表内容必须成对启用/关闭 scissor，避免展开内容越过列表边界。
- CACHE_ORDER 初始排序仍要绘制表头边框，只隐藏错误位置的排序箭头。
- Tooltip 最后提交，顺序为背景、图标、文字，避免光束或原生配方组件覆盖。
- HUD 开关必须同时重新注册或移除当前 `MaterialListHudRenderer`；只改布尔值不会出现面板。
- 切换列表、刷新缓存或替换 MaterialList 后，要让 InfoHud 指向新的渲染器。
- 禁用图标统一采用正常物品模型加中性灰遮罩；不要恢复同步 framebuffer 读回方案。
- 高版本 Litematica 自带的两个子材料导出按钮会和 LMLP 重叠，只精确移除冲突按钮，
  保留 Litematica 普通导出和 LMLP 自己的 XLSX 功能。

## 材料计算和大型整合包

- 最小子材料按每帧预算分段计算，并在普通材料页后台预热。
- 后台任务只能发布签名仍匹配的完整结果，不能显示旧任务的部分结果。
- 缓存签名包含物品、数量、JEI runtime、配方、配置和库存快照。
- JEI 查询设置截止时间；循环配方和节点数设置预算，避免第一次进入卡死。
- 优化前后必须逐项比较结果，不能只比较总行数。
- 配方产出大于 1 时，材料数量和合成次数按 output count 缩放。
- 深层循环只沿单输入转换继续检查；多输入配方可能把斧头、容器等工具暴露为输入，
  沿完整链追踪会产生“木板 → 去皮原木 → 斧头 → 木板”的假循环。
- 原木、木材、菌柄、菌核和竹块是木材链终点，不继续解析剥皮配方及工具槽。
- 煤炭块、白色床和选择组不能因为替代或可逆配方被错误拆分。
- 新原木从当前注册表/标签动态枚举，并检查候选 Tooltip、图标循环和点击选择。

## JEI 集成

- v1.8.0 已从 REI 迁移到 JEI；1.21.4/1.21.5 不参与 v1.8.x 构建。
- 始终用目标版本 JEI 校对布局、Tooltip、配方跳转和“移动到容器中”。
- 每个分类、每条配方分别隔离异常；一个第三方插件的坏 layout 不能清空已有配方。
- JEI runtime 可用或失效时，同时清查询缓存、配方树缓存和最小子材料构建缓存。
- 1.21.1 的 Litematica 0.19.60 已从 Modrinth 下架；编译使用 API 兼容的 0.19.61，
  `fabric.mod.json` 运行时兼容仍以 0.19.60 为基准。

## 发布前最低验收

- [ ] 热键绑定后重置按钮立即亮起，重置后立即变灰。
- [ ] 原点光束穿方块并位于蓝图、云层、天气和粒子上方。
- [ ] 靶心、注视标签、表头边框、HUD 和 Tooltip 正常。
- [ ] 首次进入最小子材料页不会长时间卡死，切页不被阻塞。
- [ ] 原木、煤炭块、白色床、循环配方和多产出配方结果正确。
- [ ] JEI 页面、Tooltip、配方跳转和移动按钮正常。
- [ ] JAR 的 Minecraft 版本、Java class major、Mixin、Git Commit 和 SHA-256 正确。
- [ ] 实例 `mods` 中只保留一个有效 LMLP JAR。
